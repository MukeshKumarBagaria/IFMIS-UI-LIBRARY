# Typography

The IFMIS typography system is the single source of visual truth for all text in the product. It mirrors the Figma design library 1:1 — designers and developers use the same names for the same things.

> **Rule of thumb:** never set `font-size`, `font-weight`, `line-height`, or `color` on text manually. Always use `<Heading>` or `<Text>`. If you find yourself wanting a size that doesn't exist, talk to the design team — don't invent one.

> All examples below are plain JavaScript / JSX — no TypeScript required.

---

## Font family

`Google Sans Flex` with safe fallbacks (`Google Sans`, `Inter`, `system-ui`).

Loaded automatically when you import `@ifmis/ui/styles.css`. If your environment cannot load Google Sans Flex (license restrictions), the cascade falls back to Inter and then the OS font — layout will not break.

To self-host the font, drop the woff2 files in your app's `public/fonts/` and add an `@font-face` block. The library does not bundle the font file.

## When to use what

| You need... | Use | Figma name |
| --- | --- | --- |
| A page or section title | `<Heading level={1..6}>` | H1–H6 |
| Default paragraph text | `<Text>` | b3 (16 Regular) |
| A form label or table cell label | `<Text size="xs" weight="medium">` | B4 (14 Medium) |
| A card title (not page-level) | `<Heading level={4}>` | H4 |
| Helper / hint text under a field | `<Text size="xs" color="secondary">` | b4 secondary |
| A stat number (dashboard tile) | `<Heading level={2}>` | H2 |
| Long-form content (rare in govt apps) | `<Text size="sm">` | b3 |

---

## Headings

Six levels. SemiBold (600). Line-height `100%` (no extra leading — headings sit tight).

```jsx
import { Heading } from "@ifmis/ui";

<Heading level={1}>Dashboard</Heading>
<Heading level={2}>Active Beneficiaries</Heading>
<Heading level={3}>This Month</Heading>
```

### Decouple semantics from appearance

Sometimes you need an `<h2>` that looks like an H4 (e.g. inside an accordion). Use `as`:

```jsx
<Heading level={4} as="h2">Section title</Heading>
```

The DOM gets `<h2>` (correct for screen readers and SEO), the styling gets H4.

## Body text

Body text has **3 axes** you can vary: `size`, `weight`, `color`. Everything else (font family, line height, letter spacing) is locked.

```jsx
<Text>Default — size sm, weight regular, color primary</Text>
<Text size="md" weight="medium">Larger label</Text>
<Text size="xs" color="secondary">Helper text under a field</Text>
```

### Size scale

| Prop | Pixels | Line-height | Figma equivalent |
| --- | --- | --- | --- |
| `size="lg"` | 20 | 30 | B1 / b1 |
| `size="md"` | 18 | 27 | B2 / b2 |
| `size="sm"` (default) | 16 | 24 | B3 / b3 |
| `size="xs"` | 14 | 21 | B4 / b4 |

### Weight

| Prop | CSS weight | When |
| --- | --- | --- |
| `weight="regular"` (default) | 400 | Paragraph copy, descriptions |
| `weight="medium"` | 500 | Labels, form field labels, emphasis within sentences |
| `weight="semibold"` | 600 | Strong emphasis (rare in body — usually means you wanted a Heading) |

### Color

| Prop | Hex | When |
| --- | --- | --- |
| `color="primary"` (default) | `#2E2E2E` | The content the user actually reads |
| `color="secondary"` | `#595959` | Captions, helper text, metadata, timestamps |
| `color="muted"` | theme token | Disabled-adjacent, placeholders, low-priority info |
| `color="inherit"` | — | When the parent already sets a color (e.g. inside a colored badge) |

## Polymorphic rendering — the `as` prop

Both components accept `as` to control the rendered tag without changing visuals.

```jsx
<Text as="span">inline</Text>
<Text as="label" htmlFor="email">Email address</Text>
<Heading level={3} as="h1">Looks H3, is semantically H1</Heading>
```

---

## Do / Don't

### Do

✅ Use `<Heading>` for every heading, `<Text>` for every body.
✅ Use semantic levels in document order: H1 → H2 → H3.
✅ Override the tag with `as` when semantics and visuals diverge.
✅ Use `color="secondary"` for any "supporting" information.

### Don't

❌ Don't use raw `<h1>`, `<p>`, or `<span>` with hand-rolled Tailwind classes for text styling.
❌ Don't apply `text-3xl`, `font-bold`, or `text-gray-700` directly to text — those are escape hatches that break consistency.
❌ Don't skip heading levels (H1 then H3) for visual reasons — use `as` instead.
❌ Don't hardcode hex codes for text color anywhere.

## Migration cheat-sheet

If your module uses native tags today, here is the direct mapping:

| Before | After |
| --- | --- |
| `<h1 className="text-3xl font-bold">` | `<Heading level={1}>` |
| `<h2 className="text-2xl font-semibold">` | `<Heading level={2}>` |
| `<p className="text-base">` | `<Text>` |
| `<p className="text-sm text-gray-600">` | `<Text size="xs" color="secondary">` |
| `<label className="text-sm font-medium">` | `<Text as="label" size="xs" weight="medium">` |

A codemod (`@ifmis/codemod typography`) is on the roadmap for bulk migration.

## Props

### `<Heading>`

| Prop        | Type                            | Default        | Notes                                              |
| ----------- | ------------------------------- | -------------- | -------------------------------------------------- |
| `level`     | `1 \| 2 \| 3 \| 4 \| 5 \| 6`    | `1`            | Visual size (maps to `text-h1`…`text-h6`).         |
| `as`        | `ElementType`                   | `h{level}`     | Rendered tag — decouple semantics from appearance. |
| `align`     | `left \| center \| right`       | `left`         | Text alignment.                                    |
| `className` | `string`                        | —              | Merged onto the element.                           |
| `children`  | node                            | —              | The heading text.                                  |

### `<Text>`

| Prop        | Type                                       | Default    | Notes                                          |
| ----------- | ------------------------------------------ | ---------- | ---------------------------------------------- |
| `size`      | `xs \| sm \| md \| lg`                     | `sm`       | 14 / 16 / 18 / 20px (line-height 1.5).         |
| `weight`    | `regular \| medium \| semibold`            | `regular`  | 400 / 500 / 600.                               |
| `color`     | `primary \| secondary \| muted \| inherit` | `primary`  | `inherit` keeps the surrounding colour.        |
| `align`     | `left \| center \| right`                  | `left`     | Text alignment.                                |
| `truncate`  | `boolean`                                  | `false`    | Single-line ellipsis overflow.                 |
| `as`        | `ElementType`                              | `p`        | Use `"span"` for inline, `"label"` for fields. |
| `className` | `string`                                   | —          | Merged onto the element.                        |
| `children`  | node                                       | —          | The body copy.                                 |

Both forward all native attributes of their rendered element (`id`, `aria-*`,
`data-*`, `ref`, event handlers…). The CVA recipes are also exported as
`headingVariants` / `textVariants` for composing class names elsewhere.

## Accessibility

- **Keep the heading outline correct.** Pick `level` for looks, but set `as` so
  the document's `h1`→`h6` order stays logical — screen-reader users navigate by
  it. Never skip levels just to get a size.
- Each page should have exactly one `h1` (`<Heading level={1}>` or
  `<Heading as="h1">`).
- For non-heading emphasis use `<Text as="span" weight="semibold">` rather than
  a heading tag, so you don't pollute the outline.
- Colour is decorative only — don't rely on `color="secondary"` / `"muted"`
  alone to convey meaning, and check contrast when placing text on tinted
  surfaces (the tokens meet AA on white).

## Responsive behaviour

The type scale is fixed by design tokens, so text is consistent across the
product. When you need a heading to step down on small screens, switch the
`level`/size by breakpoint rather than overriding `font-size`:

```jsx
{/* Visual size only — keep the semantic heading level correct via `as` */}
<Heading as="h1" className="text-h3 md:text-h1">Dashboard</Heading>
```

For body copy, constrain the **measure** (line length) for readability instead
of changing the size:

```jsx
<Text className="max-w-prose">Long-form explanatory copy…</Text>
```

Avoid inventing sizes — if a breakpoint needs a step that isn't in the scale,
raise it with the design team.
