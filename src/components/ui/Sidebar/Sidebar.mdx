# Sidebar

The IFMIS primary navigation rail. One component composes three regions
(search + worklist, assigned-modules, sub-modules menu) and stays **purely
presentational** — every selection is driven by props you own.

> **Rule of thumb:** the Sidebar never talks to the router, never persists
> the active module, never owns search state. Wire those to your app's
> store and pass the values down.

---

## Anatomy

```
┌───────────────────────────────────┐
│  ┌─────────────────────────────┐  │  ← top card (rounded, purple border)
│  │  [ 🔍 Search menu... ] [⇆] │  │    search input + collapse button
│  │  [ 💼 Worklist        12 ▸ ]│  │    worklist pill (optional)
│  └─────────────────────────────┘  │
│                                   │
│  ┌─────────────────────────────┐  │  ← body card (rounded, purple border)
│  │  Assigned Modules           │  │    section title
│  │  ┌──────┐  ┌──┐ ┌──┐       │  │    active card (large) + 2×2 inactive grid
│  │  │ HRMS │  │··│ │··│       │  │    + overflow chevron when > maxInlineInactive
│  │  └──────┘  └──┘ └──┘  [▸] │  │
│  │─────────────────────────────│  │
│  │  Sub-Modules                │  │    section title (customisable via menu.title)
│  │  ┌──────────────────────┐   │  │
│  │  │ Dashboard            │   │  │    leaf row
│  │  │ HRMS              ▸  │ ──┼──┼──→ right-side card (cascades for nesting)
│  │  │ Vouchers     [12] ▸  │   │  │    badge shown when MenuNode.badge is set
│  │  └──────────────────────┘   │  │
│  │─────────────────────────────│  │
│  │  (🎧) Help Desk             │  │    help links (optional, via `help`)
│  │  (?)  Help                  │  │    pinned to the bottom of the body card
│  └─────────────────────────────┘  │
│                                   │
│  [footer — any ReactNode]         │  ← optional footer slot
└───────────────────────────────────┘
```

| Region | Component | Notes |
|---|---|---|
| Frame | `Sidebar` | `279px` column, `24px` gap between cards |
| Search | `SidebarSearch` | Pill input, controlled via `value`/`onChange` |
| Collapse toggle | `SidebarCollapseButton` | Circular purple button |
| Worklist pill | `WorklistButton` | Orange pill + optional count badge + custom label |
| Modules section | `AssignedModules` | Active card + 2×2 inactive grid + overflow popover |
| Module cards | `ActiveModuleCard` / `InactiveModuleCard` | Gradient + icon per module |
| Menu | `SidebarMenu` | Top-level pill rows + cascading right-side cards |
| Submenu cards | *(internal)* `SubmenuCard` | Floating right-side cards, cascade to any depth |
| Help links | `SidebarHelpLinks` | "Help Desk" / "Help" links at the bottom of the body card |

## Quickstart

```jsx
import { useState } from "react";
import { Sidebar } from "@ifmis/ui";

const MENU = [
  { id: "dashboard", label: "Dashboard" },
  {
    id: "hrms",
    label: "HRMS",
    children: [
      { id: "hrms.exit",  label: "Exit Management" },
      { id: "hrms.leave", label: "Leave Management" },
    ],
  },
];

const ASSIGNED = ["hrms", "e-accounts", "deposit"];

function AppShell() {
  const [activeModule, setActiveModule] = useState("hrms");
  const [activeMenu, setActiveMenu]     = useState("dashboard");
  const [search, setSearch]             = useState("");

  return (
    <Sidebar
      search={{ value: search, onChange: setSearch }}
      worklist={{ count: 12, onClick: openWorklist }}
      modules={{
        assigned: ASSIGNED,
        activeId: activeModule,
        onChange: setActiveModule,
      }}
      menu={{
        items: MENU,
        activeId: activeMenu,
        onSelect: setActiveMenu,
      }}
    />
  );
}
```

Three pieces of state, three slots. Every slot is optional — omit it to hide that section.

## Props reference

### `<Sidebar>` — top-level props

| Prop | Type | Default | Description |
|---|---|---|---|
| `search` | `SidebarSearchProps` | — | Search input config. Omit to hide the input entirely. |
| `worklist` | `WorklistButtonProps` | — | Worklist pill config. Omit to hide the pill. |
| `modules` | `AssignedModulesProps` | — | Modules section config. Omit to hide the section. |
| `menu` | `SidebarMenuProps` | — | Sub-modules menu config. Omit to hide the section. |
| `help` | `SidebarHelpProps` | — | "Help Desk" / "Help" links pinned to the bottom of the body card. Omit to hide both. |
| `collapsed` | `boolean` | `undefined` | Controlled collapse state. Omit to let the Sidebar self-manage. |
| `onCollapseToggle` | `() => void` | — | Fired when the collapse circle is clicked. Pair with `collapsed`. |
| `footer` | `ReactNode` | — | Extra content rendered below the body card. |
| `className` | `string` | — | Extra CSS class added to the outer `<aside>`. |
| `...rest` | `HTMLAttributes<HTMLElement>` | — | Any other valid `<aside>` attributes (e.g. `id`, `data-*`, `style`) are spread onto the root element. |

### `search` — `SidebarSearchProps`

The pill-shaped search input at the top of the rail.

```ts
interface SidebarSearchProps {
  value?:       string;                    // controlled value
  onChange?:    (value: string) => void;   // fires on every keystroke
  placeholder?: string;                    // default: "Search menu..."
}
```

- **Omit** the entire `search` prop to hide the input.
- The value is **automatically forwarded** to the `menu` so it enters search mode. You can also override by setting `menu.searchValue` explicitly.
- `placeholder` defaults to `"Search menu..."`.

```jsx
<Sidebar search={{ value: q, onChange: setQ }} ... />
<Sidebar search={{ value: q, onChange: setQ, placeholder: "Search HRMS..." }} ... />
```

### `worklist` — `WorklistButtonProps`

The orange pill below the search row.

```ts
interface WorklistButtonProps {
  count?:   number;        // badge shown on the right; omit to hide badge
  onClick?: () => void;    // called when the pill is clicked
  label?:   string;        // visible label; default: "Worklist"
}
```

- **Omit** the entire `worklist` prop to hide the pill.
- `count` is optional — pass it to show the badge, omit it to hide.
- `label` defaults to `"Worklist"`.

```jsx
<Sidebar worklist={{ count: 12, onClick: openWorklist }} ... />
<Sidebar worklist={{ onClick: openWorklist }} ... />                 // no badge
<Sidebar worklist={{ label: "Pending Tasks", count: 7, onClick }} ... />  // custom label
```

### `modules` — `AssignedModulesProps`

The assigned-modules section with the active card, inactive thumbnails, and overflow popover.

```ts
interface AssignedModulesProps {
  assigned:           ModuleId[];                    // ordered list of assigned module ids
  activeId:           ModuleId | null;               // selected module, or null → empty state
  onChange?:          (id: ModuleId) => void;        // called when the user switches modules
  maxInlineInactive?: number;                        // default: 3
}
```

**`assigned`** — Array of `ModuleId` strings. Usually fetched from your backend at sign-in.
The Sidebar handles any length gracefully:
- `0` modules → section is still rendered (pass modules only if you have at least one)
- `1–4` modules → fills the 2×2 grid; no overflow
- `5+` modules → shows `maxInlineInactive` thumbnails + a `▸` overflow button

**`activeId`** — The currently selected module id, or `null`.
- When `null`, the section renders the empty state (illustration + "Select Module" button).
- Setting to `null` is the correct initial state when the user hasn't picked a module yet.

**`onChange`** — Called when the user clicks an inactive thumbnail or selects from the overflow popover.

**`maxInlineInactive`** (default `3`) — How many inactive thumbnails appear inline before the overflow chevron.
- `3` → 3 thumbnails + overflow button in the 4th 2×2 slot (recommended for most cases)
- `4` → fills all 4 slots; use when you know users have at most 4 modules

```jsx
// Server-driven assignment
const { data: user } = useCurrentUser();

<Sidebar
  modules={{
    assigned: user.assignedModules,
    activeId: user.lastActiveModule,
    onChange: persistActiveModule,
  }}
/>
```

### `menu` — `SidebarMenuProps`

The sub-modules menu with pill rows and cascading right-side cards.

```ts
interface SidebarMenuProps {
  title?:               string;                    // default: "Sub-Modules"
  items:                MenuNode[];                // the menu tree
  activeId?:            string;                   // currently selected leaf id
  onSelect?:            (id: string) => void;     // fired when a leaf is clicked
  searchValue?:         string;                   // switches to search-results view when non-empty
  searchResultsTitle?:  string;                   // default: "Search results"
  searchEmptyTitle?:    string;                   // default: "No matches"
}
```

**`title`** — Section heading above the menu. Defaults to `"Sub-Modules"`.

**`items`** — Your `MenuNode[]` tree. See [Menu authoring](#menu-authoring) below.

**`activeId`** — The id of the currently active leaf. Set this to your current route.
A row is highlighted (purple) when its `id` matches `activeId` **or** its submenu card is open.

**`onSelect`** — Called when the user clicks a leaf (a node with no children).

**`searchValue`** — When non-empty the menu switches to a flat search-results view.
The `<Sidebar>` parent wires this automatically from `search.value` — you only
need to set it manually when using `<SidebarMenu>` in isolation.

**`searchResultsTitle`** — Heading shown above the results list. Default: `"Search results"`.

**`searchEmptyTitle`** — Heading shown when there are zero results. Default: `"No matches"`.

### `help` — `SidebarHelpProps`

The two support links pinned to the bottom of the body card — a purple
**Help Desk** (headset) link and a blue **Help** (question-mark) link.

```ts
interface SidebarHelpItem {
  label?:   string;        // override the visible label
  onClick?: () => void;    // open your help desk / help center
}

interface SidebarHelpProps {
  helpDesk?: SidebarHelpItem;   // omit to hide just this link
  help?:     SidebarHelpItem;   // omit to hide just this link
}
```

- **Omit** the entire `help` prop to hide both links.
- Omit one of `helpDesk` / `help` to hide only that link.
- Each link defaults its label (`"Help Desk"` / `"Help"`); override via `label`.
- In the **collapsed** rail these render as icon-only circles (no labels) at the
  bottom of the body card, keeping the same purple / blue tones.

```jsx
<Sidebar
  help={{
    helpDesk: { onClick: openHelpDesk },
    help:     { onClick: openHelpCenter },
  }}
/>

// Custom labels
<Sidebar
  help={{
    helpDesk: { label: "Support",       onClick: openSupport },
    help:     { label: "Documentation", onClick: openDocs },
  }}
/>
```

> The links are also exported standalone as `SidebarHelpLinks` for bespoke layouts.

### `collapsed` / `onCollapseToggle`

The Sidebar has two modes: **expanded** (`279px`) and **collapsed** (`120px`).

**Uncontrolled** — omit both props; the Sidebar self-manages the toggle:

```jsx
<Sidebar ... />  // click the ⇆ circle to collapse/expand; state is internal
```

**Controlled** — pass both props to persist the state externally:

```jsx
const [collapsed, setCollapsed] = useLocalStorageState("sidebar.collapsed", false);

<Sidebar
  collapsed={collapsed}
  onCollapseToggle={() => setCollapsed((v) => !v)}
  ...
/>
```

In the collapsed rail, every slot mirrors its expanded counterpart:
- Search icon → expands sidebar and focuses the search input
- Compact worklist button (icon + optional count)
- "Assigned Modules" label + active module thumbnail + `▸` expand button
- Sub-modules toggle (hamburger + chevron)
- Help links → icon-only circles (purple headset / blue question mark)

<Sidebar

## Menu authoring

### `MenuNode` shape

```ts
interface MenuNode {
  id:        string;            // stable key — use a route path if you have one
  label:     string;            // visible text in the row
  badge?:    string | number;   // optional counter; omit to hide
  children?: MenuNode[];        // presence makes the row a parent (opens a right-side card)
}
```

### Leaves vs. parents

- **Leaf** (`children` absent or empty array) — clicking calls `onSelect(id)`.
- **Parent** (`children` present) — clicking opens a floating right-side card; the row's id is still passed to `onSelect`.

### Nesting depth

Children can be nested to **unlimited depth**. Each level opens a card to the right of the previous one.

```js
{ id: "hrms", label: "HRMS", children: [
  { id: "hrms.pay", label: "Pay Related", children: [
    { id: "hrms.pay.salary",  label: "Salary"  },
    { id: "hrms.pay.bonus",   label: "Bonus"   },
    { id: "hrms.pay.arrears", label: "Arrears" },
  ]},
]}
```

### Badges

Pass a `string` or `number` to `badge` for a counter. The pill colour inverts automatically
based on the active state (purple-on-white inactive, white-on-purple active).

```js
{ id: "vouchers", label: "Vouchers", badge: 12, children: [
  { id: "vouchers.pending", label: "Pending", badge: 8 },
  { id: "vouchers.approved", label: "Approved" },
]}
```

### Active state logic

The **top-level** row is highlighted when:
- its `id === activeId`, **or**
- its submenu card is currently open

Deep leaves inside submenu cards are highlighted when their `id === activeId`.

You typically set `activeId` to the **leaf** the user navigated to — nothing else needs to be tracked.

### Tip — use route paths as ids

```js
{ id: "/hrms/exit-management", label: "Exit Management" }
```

Then wire `onSelect` to your router and `activeId` to the current pathname:

```jsx
<Sidebar
  menu={{
    items: MENU,
    activeId: pathname,
    onSelect: (id) => navigate(id),
  }}
/>
```

## Search behaviour

The built-in search uses a **case-insensitive substring match** that walks the entire menu tree:
- A query matching a **leaf label** → that leaf appears.
- A query matching a **parent label** → every leaf under that parent appears.
- Each result shows a breadcrumb of the ancestor path (e.g. `HRMS / Pay Related`).
- The matched substring is highlighted inside the label.

The search is scoped to the tree you pass in `menu.items` — it never searches across modules.
When the user switches modules, update `menu.items` to the new module's tree.

## Wiring patterns

### 1. Bind selection to your router

```jsx
import { useNavigate, useLocation } from "react-router-dom";

const navigate = useNavigate();
const { pathname } = useLocation();

<Sidebar
  menu={{
    items: MENU,
    activeId: pathname,
    onSelect: (id) => navigate(id),
  }}
/>
```

### 2. Switch the menu when the module changes

Each module gets its own menu tree. Keep them in a record keyed by `ModuleId`:

```js
const MENUS = {
  hrms:         HRMS_MENU,
  "e-accounts": E_ACCOUNTS_MENU,
  budget:       BUDGET_MENU,
};
```

```jsx
<Sidebar
  modules={{ assigned, activeId: activeModule, onChange: setActiveModule }}
  menu={{
    items: MENUS[activeModule] ?? [],
    activeId,
    onSelect,
  }}
/>
```

### 3. Server-driven module assignment

```jsx
const { data: user } = useCurrentUser();

<Sidebar
  modules={{
    assigned: user.assignedModules,   // ModuleId[]
    activeId: user.lastActiveModule,  // ModuleId | null
    onChange: persistActiveModule,
  }}
/>
```

### 4. Persist collapse state across reloads

```jsx
const [collapsed, setCollapsed] = useState(
  () => localStorage.getItem("sidebar.collapsed") === "true",
);

<Sidebar
  collapsed={collapsed}
  onCollapseToggle={() => {
    const next = !collapsed;
    setCollapsed(next);
    localStorage.setItem("sidebar.collapsed", String(next));
  }}
  ...
/>
```

### 5. Auto-collapse on small viewports

```jsx
const isSmall = useMediaQuery("(max-width: 1024px)");
const [collapsed, setCollapsed] = useState(false);

<Sidebar
  collapsed={isSmall || collapsed}
  onCollapseToggle={() => setCollapsed((v) => !v)}
  ...
/>
```

## The module catalogue

`MODULES` and `MODULE_LIST` are the canonical source of truth for IFMIS module metadata
(icon, gradient, label, text colour, icon-badge shadow). Use them in product code whenever
you need the same visuals outside the Sidebar — breadcrumbs, page headers, landing-screen
module pickers.

```ts
interface ModuleDef {
  id:         ModuleId;   // stable key used for selection / routing
  label:      string;     // visible name shown on the active card
  Icon:       Icon;       // Phosphor icon component
  gradient:   string;     // linear-gradient used by the active + inactive cards
  textColor?: string;     // label + icon colour; defaults to "#4B3960"
  shadow:     string;     // icon-badge drop shadow — a soft tint of the module colour
}
```

```jsx
import { MODULES, MODULE_LIST } from "@ifmis/ui";

// Look up a single module
const { Icon, label, gradient, textColor, shadow } = MODULES["hrms"];

// Iterate all modules
MODULE_LIST.map((m) => (
  <li key={m.id} style={{ background: m.gradient, color: m.textColor }}>
    <span style={{ boxShadow: m.shadow }}>
      <m.Icon size={24} />
    </span>
    {m.label}
  </li>
));
```

## Accessibility

- The Sidebar is wrapped in `<aside aria-label="Primary navigation">`.
- The collapsed rail uses `aria-label="Primary navigation (collapsed)"`.
- Submenu rows carry `role="menuitem"` and cards carry `role="menu"`.
- Submenu triggers expose `aria-expanded` and `aria-haspopup="menu"`.
- All interactive elements are focusable via Tab and operable via Enter / Space.
- `Escape` closes the currently open submenu card (both in expanded and collapsed states).
- The active leaf carries `aria-current="page"`.
- The collapse button exposes `aria-label="Collapse sidebar"` / `"Expand sidebar"` and `aria-pressed`.
- Inactive module thumbnails carry an `aria-label` with the module name.
- The overflow popover opens as a `role="dialog"` with `aria-label="All assigned modules"`.

## When **not** to use Sidebar

- **Single-module product** — if there is only one module and no sub-navigation, render
  your own top tabs or breadcrumbs. The full rail will feel disproportionate.
- **Dashboard without modules** — use just `<SidebarMenu />` on its own, without the `modules` slot.
- **Mobile / phone viewports** — prefer hiding the rail behind a drawer (your app owns the
  drawer shell) and rendering the **full** Sidebar inside it, rather than showing the collapsed
  rail permanently. The 120px collapsed width is still too wide for small phones.
